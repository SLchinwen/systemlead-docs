## 定價治理辦法

> **文件性質：非 SSOT（說明文件）**  
> 本文件為定價治理框架之內部參考。具體價格、折扣、核准權限，以公司正式公告為準。

---

## 0. 文件基本資訊

- **文件用途**
  - 對外：提供「為何採年度張數計費」的清楚說法，協助業務/客服一致溝通
  - 對內：建立定價治理框架，支援年度續約、定價檢討、折扣策略與競品對標
- **適用對象**：業務、客服、產品、財務、營運、管理層
- **適用範圍**：e首發票加值中心之標準服務（年度張數方案）與續約定價（不含客製專案報價）
- **產品命名與級距資料**：方案代碼、顯示名稱與年度張數級距錨點，以 [pricing-tiers.yaml](./pricing-tiers.yaml) 為單一維護來源；級距計算公式見 [級距公式與維護說明](./pricing-tier-formula.md)。

## 0.3 年度張數級距公式（主約年費計算）

- **輸入**：年度張數，以**千張**為單位（例：1 = 1,000 張、240 = 240,000 張）。
- **輸出**：年費（元）。**非**單價×數量，採**分段線性內插**（相鄰錨點間線性計算），體積愈大單張邊際愈低。
- **錨點來源**：[pricing-tiers.yaml](./pricing-tiers.yaml) 各方案之 `tierAnchors`。
- **公式與範例**：見 [級距公式與維護說明](./pricing-tier-formula.md)。  
- 系統實作：依 YAML 錨點表與公式算出費用，無需在程式內寫死級距表。

## 0.5 報價前必區分：資料來源與取號規則

> 報價與定價治理時，**須先區分客戶之資料來源與取號規則**，再選擇對應方案與級距。  
> 本節為業務／客服對客戶說明「為何訂單版／發票版／混合」及「為何發票版費用較高」之依據，並可引用於報價預算書以避免錯誤與爭議。

### 訂單版（以「訂單」為核心）

| 項目 | 說明 |
| ---- | ---- |
| **資料來源** | 客戶提供 **訂單資料**，由 e首發票負責取號與開立，並回傳發票號碼 |
| **取號責任** | 加值中心集中取號，**較不會有重複取號問題** |
| **適用情境** | 訂單 EXCEL 匯入、網站接單、系統 API 串接、電商、需訂單查詢與追蹤 |
| **報價對應** | 訂單版方案、對應訂單版級距與開通費 |

### 發票版（以「已取號發票」為核心）

| 項目 | 說明 |
| ---- | ---- |
| **資料來源** | 客戶端 **已取號之發票**，上傳至加值中心負責記錄、上傳、管理與申報 |
| **取號責任** | 客戶或前端系統取號，加值中心負責上傳與治理 |
| **風險與配套** | 前端可能出現 **重號、傳送失敗重傳、逾期、稽核問題**，需更多驗證與配套 SOP 治理以避免錯誤 |
| **費用說明** | 因驗證、重送、稽核與錯誤防呆成本較高，**發票版費用較訂單版高** |
| **報價對應** | 發票版方案、對應發票版級距與開通費 |

### 混合使用（訂單版 + 發票版並用）

| 項目 | 說明 |
| ---- | ---- |
| **定義** | 營業人同時使用「以訂單為核心」與「以已取號發票為核心」之情境 |
| **報價規則** | **一律以發票版取號報價**（依發票版方案、級距與開通費） |
| **理由** | 避免取號責任與風險切割不清、重號與稽核爭議，並與配套 SOP 治理一致 |

### 對外說明時可引用（避免錯誤）

- 「訂單版由我們集中取號，重複取號風險低；發票版是貴司或前端已取號再上傳，我們負責上傳與申報，需較多驗證與稽核配套，所以費用較高。」
- 「若貴司會同時用訂單與已取號發票兩種方式，我們一律以發票版方案報價，確保取號與稽核規則一致。」

## 1. 對客戶的說法：為何採「年度張數」計費

### 1.1 一句話版本（最常用）

> **年度張數計費**讓您用「可預算、可控管」的方式使用雲端電子發票服務：費用與用量清楚對應、年度可調整級距，避免一開始買太多或臨時超量造成成本失控。

### 1.2 30 秒版本（業務口徑）

- 我們的服務是雲端加值中心，不是裝在貴司機器上
- 平台提供 **合規上傳、查詢管理、稽核機制、通知與整合能力**，需要全年維運與資安控管
- 用「年度張數」的方式：
  1. **好做預算**：一開始就知道今年可用張數與費用
  2. **好調整**：續約時可依過去用量調整下一年度級距
  3. **好擴充**：張數不足可加購，不會影響既有作業
  4. **更公平**：不同客戶依實際用量與服務強度付費

### 1.3 長版（給採購/財會/IT 的理由）

年度張數計費是 SaaS（雲端服務）常見且合理的計價方式，主要原因：

1. **成本結構與風險對齊**
   - 電子發票服務需要全年穩定維運（排程、稽核、重送、異常追蹤）、合規處理、資安監控、備援與日誌
   - 這些成本與「是否有使用、使用多少」高度相關，年度張數能反映服務強度

2. **預算可控與內控友善**
   - 客戶可在年度初或合約期初就確定年度可用量，符合採購與預算編列

3. **彈性續約與避免浪費**
   - 續約時客戶可登入帳號，依歷史使用量調整下一年度方案
   - 需要時再加購張數，降低一次買太多造成浪費

4. **適配不同商業型態**
   - 電商、POS、ERP 串接、機台等模式，每張發票帶來的系統負載、通知與查詢頻率不同
   - 年度張數級距＋加購模組可讓客戶用「組合」挑到最合適的成本結構

## 2. 續約與客戶自助調整（對外流程描述）

### 2.1 續約時客戶能做什麼

- 客戶可自行登入後台，查看：
  - 去年/今年使用張數
  - 常用模組與加購項目
  - 通知/中獎等儲值消耗
- 依據歷史使用計畫，調整：
  - 次年度張數級距
  - 需要續購的模組（如蝦皮 API、雲列印、媒體申報、B2B 等）

### 2.2 我們提供的建議（標準引導）

- 若使用量穩定：建議選「接近但略高於去年用量」的級距（保留成長空間）
- 若有季節性高峰：建議以高峰月推估年度量，或搭配加購張數
- 若年中才導入：可依政策提供「方案延展」或「較高級距更划算」的建議（依當年度活動規則）

## 3. 定價治理辦法（內部）

### 3.1 治理目標

- **市場合理**：價格能被客戶理解與接受
- **競爭力**：在同級加值中心/替代方案下具可比優勢
- **可持續**：覆蓋雲端維運、資安、客服與產品迭代成本
- **可治理**：折扣、方案、續約與加購規則可追蹤、可審核、可復盤

### 3.2 定價範圍與分層

定價應分成三層，避免混在一起造成誤解：

| 層級 | 說明 |
| ---- | ---- |
| **主約** | 年度張數級距（核心收入）；年費由 [級距公式](./pricing-tier-formula.md) 依輸入千張算出，錨點維護於 [pricing-tiers.yaml](./pricing-tiers.yaml) |
| **選購模組** | 功能型年費 / 一次性開通費 |
| **儲值型服務** | 通知點數、中獎列印張數等（按量消耗） |

並明確標示：

- **標準服務**：線上下單、年費/方案價
- **專案服務**：需出具工時估算與專案報價

### 3.3 年度定價檢討節奏（建議）

| 頻率 | 內容 |
| ---- | ---- |
| **每年 1 次** | 年度定價檢討（建議 9–11 月完成，銜接次年度續約與首購活動） |
| **每季 1 次** | 輕量檢視（客訴/折扣使用率/轉換率/成本與異常） |
| **每月** | 營運儀表板追蹤（不用改價，但要看趨勢） |

### 3.4 定價檢討輸入資料（必備）

#### A. 市場與競品

- 同級加值中心：年費/張數級距/最低門檻、模組收費結構、首購/續約優惠策略
- 替代方案：自建 Turnkey + 維運成本（人力、資安、備援、合規風險）

#### B. 內部營運

- 客戶分級用量分布（P50/P75/P90）
- 續約率、流失率、客單價（ARPA）
- 張數超用與加購比例
- 模組採用率（attach rate）
- 客服工單量與類型（每 1,000 張工單數）
- 平台成本（雲資源、訊息成本、支援人力）

#### C. 價格感受與銷售效率

- 轉換率（試算→下單）
- 折扣使用率與折扣後淨收益
- 價格阻力原因分類（太貴/看不懂/怕綁約/不確定用量）

### 3.5 決策規則（建議採用）

#### 主約級距的調整

- 原則：小步快跑；避免大幅波動造成續約阻力
- 依「成本變動 + 競品對標 + 續約敏感度」三者綜合評估

#### 模組定價

- 以「價值清楚、可獨立購買」為原則
- 明確標示一次性（開通/設定）與年費（維運/平台資源）

#### 折扣治理

- 折扣只作為「活動策略」與「渠道策略」
- 每個折扣需有：適用條件、期間、上限、核准人
- 折扣不得破壞主約價格錨定（避免客戶把折扣當常態）

## 4. 對外定價說明的標準結構

業務/客服對外溝通時，應一致依下列結構說明：

1. 我們是雲端加值中心（非裝機），平台全年維運與合規/資安由我們承擔
2. 主約用年度張數：好預算、好調整、好擴充
3. 張數之外，可加購需要的模組：電商整合、POS、雲列印、通知、中獎服務等
4. 續約時可依歷史用量自助調整；不足可加購；年中導入可依政策提供更合適方案

## 5. 未來改善方向

### 5.1 定價與方案優化

- 更清楚的級距建議器：以歷史用量 + 成長率，推薦最適級距
- 超用保護機制：接近用量上限時主動提醒或建議加購
- 模組打包策略：針對電商/門市/機台客群提供 bundle
- 續約頁面可視化：把「去年用量、今年建議、價差」一次呈現

### 5.2 競爭力提升

- 以「合規安心 + 穩定上線 + 維運省心」作為定價價值主張
- 對外提供「自建成本對比」簡表（非攻擊式，偏風險與人力成本說明）
- 強化 Azure 雲端架構與資安合規敘述（SLA、備援、權限隔離、日誌）
