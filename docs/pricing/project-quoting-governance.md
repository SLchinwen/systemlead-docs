## 專案報價治理辦法

> **文件性質：非 SSOT（說明文件）**  
> 本辦法為專案與混合報價之內部治理規範。具體價格與核准權限以公司公告為準。

---

## 一、目的與適用範圍

本辦法用於規範 **電子發票加值中心** 對外提供 **專案型** 與 **混合型** 報價時之原則、判斷邏輯、作業程序與文件標準。

**適用情境：**

- 加值中心外掛／延伸之專案型開發報價
- **非線上報價**之 e首發票客戶
- **混合報價**：同時含客製收費與發票模組年費

**對照：** 標準服務（線上下單）→ 依 [定價治理辦法](./pricing-governance.md) 與 [業務報價 SOP](./sop-sales-quoting.md)。

---

## 二、報價前必區分：資料來源與取號規則

專案／混合報價涉及發票模組時，**須先區分客戶之資料來源與取號規則**，再填寫報價預算書之標準服務區塊。

| 類型 | 取號責任與風險 | 報價對應 |
| ---- | -------------- | -------- |
| **訂單版** | 以訂單為核心，加值中心取號與開立，重複取號風險低 | 訂單版方案、級距、開通費 |
| **發票版** | 以已取號發票為核心，上傳／管理／申報，需較多驗證與稽核配套，費用較高 | 發票版方案、級距、開通費 |
| **混合使用** | 營業人同時使用訂單與發票上傳 | **一律以發票版取號報價** |

詳見 [定價治理辦法 - 資料來源與取號規則](./pricing-governance.md#05-報價前必區分資料來源與取號規則)。  
報價預算書中可引用該節說明，避免客戶誤解取號責任與為何發票版費用較高。

---

## 三、報價類型治理原則

### 3.1 標準服務 vs 專案服務 vs 混合報價

| 類型 | 定義 | 報價方式 |
| ---- | ---- | -------- |
| **標準服務** | 既有雲端模組、多客戶共用 | 線上下單、年費制，不需報價預算書 |
| **專案服務** | 客製開發、僅此客戶使用 | 須出具報價預算書、工時估算 |
| **混合報價** | 專案客製 + 標準發票模組 | 報價預算書分區塊列出兩者 |

### 3.2 混合報價必備結構

報價預算書須 **分區塊、分項列出**：

| 區塊 | 內容 |
| ---- | ---- |
| **A. 標準服務（發票模組）** | 年度張數主約、選購模組、儲值（年費／一次性） |
| **B. 專案客製** | 客製項目、工時、單價、金額 |
| **C. 責任切割與排除事項** | 加值中心 vs 客戶責任、Not In Scope |

總計與付費時點須清楚標示。

---

## 四、報價作業程序

1. **需求判斷**：是否影響其他客戶？需新增程式？僅此客戶使用？任一為「是」→ 專案
2. **責任切割**：加值中心負責範圍、客戶負責範圍、排除事項
3. **模組拆解**：不得以「整包開發」為唯一項目，須拆解功能模組

---

## 五、工時估算治理（專案必備）

- 專案報價 **必須附帶工時估算表**，不得僅提供總金額
- 欄位：功能模組名稱、工作內容摘要、預估工時（人時）
- 工時含：分析設計、開發、測試、文件與溝通
- 報價金額 = 工時 × 單價（單價依公司標準）

---

## 六、報價預算書標準結構

### 6.1 專案報價（純專案）

1. 專案背景與定位
2. 雲端與機房說明
3. 專案目標與範圍（含排除事項）
4. 功能模組與工時估算表
5. 報價金額與計算方式
6. 年費與後續維運（如有）
7. 責任切割與前提條款

### 6.2 混合報價（專案 + 發票模組）

除 5.1 外，須額外包含：

1. **標準服務費用彙總**：主約級距、選購模組、儲值
2. **專案客製費用彙總**：功能模組、工時、單價、小計
3. **總計與付費時點**：一次性 vs 年度、分期說明

---

## 七、對外治理語言

- 「本項為加值中心**標準服務**，採年費制」
- 「本需求屬於**加值中心外掛專案**，需另行專案報價」
- 「系統運行於 e首發票 Azure 雲端平台，非建置於客戶指定主機」
- 「帳務、ERP、採購邏輯不屬於本專案範圍」
