本預算書依下列架構說明，方便貴司評估與採購決策：

## 二、專案背景與定位說明

### 2.1 e首發票加值中心既有服務模式

e首發票為財政部核准之**電子發票加值服務中心**，採用雲端化服務架構，提供營業人以「線上申請、線上下單」方式，直接採購並啟用標準化之電子發票模組服務。

一般情況下：

- 加值中心標準服務 **不需另行提出專案預算書**
- 客戶可依實際需求，直接於線上完成服務採購與啟用

### 2.2 本案性質說明（加值中心外掛專案）

本案屬於：

> **加值中心標準模組以外之「外掛專案開發」**

係因貴司有特定系統整合與資料治理需求，需於既有 e首發票雲端服務架構上，額外延伸專屬 API 與管理機制，故以「專案方式」進行評估與開發，並提出本報價預算書。

📌 本專案**非**：

- 客戶自行建置電子發票系統
- 客戶指定主機或指定機房之系統開發

所有專案功能皆運行於 **e首發票加值中心既有雲端平台** 之上。

---

## 三、加值中心雲端服務與機房架構說明

### 3.1 雲端平台架構

e首發票加值中心採用 **Microsoft Azure 雲端平台** 作為核心服務架構，所有電子發票相關系統皆部署於 Azure 合規機房環境中。

主要特色：

- 採雲端高可用度架構，具備自動備援與容錯設計
- 服務可用度高，避免單點故障
- 無需客戶自行建置或維運伺服器

### 3.2 資安與法規符合性

e首發票平台已符合電子發票相關法規與資安要求，包含：

- 財政部電子發票加值中心規範
- 雲端環境資安控管與權限隔離
- 資料存取與操作紀錄留存

📌 本專案僅為功能延伸，不影響既有加值中心之資安與合規架構。

## 七、本專案帶來的效益與價值（銷售說明）

> 協助貴司理解選擇 e首發票加值中心的優勢，支持採購決策。

### 7.1 本專案效益

- **降低自建成本**：無需自行建置電子發票系統，避免高昂開發與維運投入
- **雲端免維運**：系統部署於 Azure 機房，貴司無需提供主機或負擔資安維運
- **合規有保障**：加值中心已符合財政部規範，專案延伸不影響既有合規架構
- **一次投入、長期使用**：專案開發為一次性投入，後續僅需年度服務費，預算可預期

### 7.2 選擇 e首發票加值中心的優勢

- **財政部核准加值中心**，服務穩定可靠
- **Azure 雲端架構**，高可用、自動備援
- **集中維運**，客戶可專注本業
- **彈性擴充**，未來可依需求加購標準模組或延伸專案

## 八、責任切割與結語

### 8.1 責任歸屬

- **加值中心負責**：雲端平台維運、專案功能開發與上線、API 文件與支援
- **貴司／系統商負責**：ERP 或內部系統串接、業務邏輯實作、端對端測試配合
- **排除事項**：帳務、會計分錄、採購單比對等，不屬於本專案範圍

### 8.2 結語

透過本專案，貴司可在既有 e首發票加值中心雲端平台上，延伸符合自身需求之電子發票整合功能，兼顧合規性、系統穩定度與長期維運彈性。

本報價預算書作為加值中心外掛專案之評估依據，實際專案內容將依雙方確認後執行。
